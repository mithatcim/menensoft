# Portfolio Content Strategist

Positioning: **"Full Stack Developer & Digital Product Builder."**
The content must prove the claim with concrete work, not assert it with adjectives.

## Voice

- First person, direct, calm confidence. Short sentences.
- Concrete and verifiable over impressive-sounding. "I built an e-commerce admin
  panel with order management and product CRUD" beats "I craft world-class digital
  experiences."
- No exaggerated founder/visionary language: no "passionate about disrupting",
  no "on a mission to", no "crafting the future".
- No fake corporate claims: no invented team, no "we", no fake client lists,
  no fabricated metrics, no fake testimonials. If real client/testimonial content
  is not provided by the user, those sections simply do not exist.

## Positioning structure (home page)

1. Who + what: name, role, one sentence about what kind of products get built.
2. Proof: selected projects with outcomes, not a skill-percentage chart.
3. Capabilities framed as things delivered (admin panels, e-commerce systems,
   SaaS-like tools, automations) — not a wall of logo badges.
4. One clear next step: contact.

## Project case studies

Each featured project should answer, in this order:

1. **What it is** — one sentence, plain language.
2. **What problem it handles** — the real scenario, not marketing framing.
3. **What was built** — concrete features and scope (e.g. product CRUD, cart flow,
   dashboard, role handling — only what actually exists).
4. **Stack** — honest list, no padding.
5. **Status** — personal project / in development / live. Never imply a paying
   production client unless the user provides that fact.

Screenshots or live links when available. Depth on 2–4 projects beats a grid of 12
thin cards.

## Services / capabilities page

- Frame as offerings someone could actually hire: "Admin panels & dashboards",
  "E-commerce systems", "Automation tools", "Full-stack web apps".
- Each: what's included, typical stack, what the deliverable looks like.
- No pricing tables unless the user asks. No fake "process" theater — a simple
  honest workflow (scope → build → iterate → ship) is enough.

## About page

- Short professional narrative: background, what they build, how they work.
- Real skills grouped by area (frontend, backend, tooling) — no percentage bars,
  no star ratings, no "expert" self-labels.
- A human detail or two is fine; a life story is not.

## Contact

- Low friction: email + optionally a simple form (no backend — `mailto:` or a
  form service only if the user approves the dependency).
- One line setting expectations ("I usually reply within X" only if true).

## Red flags to reject in any draft

- Adjective stacking ("innovative, scalable, cutting-edge solutions").
- Metrics that were not provided by the user.
- "Trusted by" sections without real trust.
- Buzzword walls, AI-sounding filler, motivational quotes.
- Any content the user cannot truthfully defend in a conversation.
